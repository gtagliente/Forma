# FT-005 — Exercise Form Visibility Fix

## Status

Requirements drafted (Frontend Analyst); pending Frontend Architect design.

## Requirements (Frontend Analyst)

### Summary

Bug fix, not a new feature and not a data-wiring problem. On `ExerciseLibrary.tsx` (`my-frontend/src/pages/ExerciseLibrary.tsx`), clicking "Nuovo Esercizio" (or a row's Edit icon) opens the create/edit form panel, which renders `ExerciseForm` (`my-frontend/src/components/Exercise/ExerciseForm.tsx`). The panel is functionally present — form fields exist in the DOM and are wired to real `createExercise`/`updateExercise` calls per FT-002 — but it is visually unreadable: no label, input, textarea, checkbox text, or button text is legible. No backend call, DTO, contract, or business/domain rule is implicated anywhere in this bug; it is a pure CSS/text-contrast defect local to this app's own presentation layer, squarely in scope for this pipeline per `CLAUDE.md` ("this app owns layout, graphics... [and can fix] broken CSS/visibility").

### Root cause, confirmed by reading both components (not assumed)

- `ExerciseLibrary.tsx` wraps the whole page, including the form panel, in `<div className="p-4 max-w-screen-xl min-h-screen bg-gray-950 text-white">` — dark background, white text, cascading to all descendants that don't set their own color.
- `ExerciseForm.tsx`'s `<form>` sets `className="bg-gray-50 p-6 rounded-lg border"` — a light gray background — but none of its `<label>`, `<input>`, `<textarea>`, or checkbox-row elements set an explicit text color of their own. They inherit the ancestor's `text-white`, producing white text on a near-white background. Only the submit `<button>` is legible, because it explicitly sets `bg-indigo-600 text-white` — dark background, white text, no inheritance problem there.
- This is exactly the tension `FT-002-exercise-wiring.md`'s Design section already documented under "Styling note, flagging a tension" and deliberately left unresolved, forwarding it to Central Architect Gate as a known, non-blocking inconsistency rather than fixing it. It was never fixed. This bug report is that same pre-existing tension now causing an actual usability failure (a form a user cannot read or use), not a new discovery — the underlying cause has been on record since FT-002.

### What "fixed" means (requirement, not implementation)

All text inside the `ExerciseForm` panel — every label ("Nome Esercizio", "Descrizione", "Gruppi Muscolari", each muscle-group checkbox's text, and the submit button's "Salva"/"Aggiungi" label), every input/textarea's typed and placeholder text, and any error state text this form already surfaces — must be legible against its own background, wherever that background ends up, consistent with the dark theme used by the rest of the app (`ExerciseLibrary.tsx` itself, `ExerciseLibraryItem.tsx`, `AppHeader.tsx`, etc.).

I am not specifying which side resolves the contrast (e.g., whether the form panel keeps a light card and gains explicit dark text, or adopts the app's existing dark-surface convention and gains explicit light text) or which exact Tailwind classes to use — that is the Frontend Architect's design call, consistent with this feature's FT-002 precedent already flagging (but not deciding) exactly this choice.

### Explicitly not a layout/redesign request

Per `CLAUDE.md`'s "one rule," I'm flagging this distinction rather than letting it blur: fixing text/background contrast so existing labels, inputs, and controls are readable is a CSS/wiring-adjacent bug fix, not a visual/layout redesign. The form's field set (`name`, `description`, `muscleGroups`), field order, structure, and the fact that it reuses `ExerciseForm` rather than a new component are all out of scope for this feature — none of that changes. If the fix the Architect/Developer settles on requires restructuring the form's JSX beyond className/color adjustments (e.g., moving fields around, changing which component renders what), that would cross into a layout change and should be flagged back rather than done silently under this bug fix.

### No backend/domain dependency

Confirmed against `../../Forma.Claude/docs/services/web-client/open-questions.md`: none of the currently-open items (training-execution-service unimplemented, cross-service JWT validation, missing CORS config, no refresh-token endpoint) touch this bug in any way. This fix requires no API contract, no data model change, and no decision from any backend service's Service Analyst — it is entirely local to `web-client`'s own CSS.

### Scope

- **In scope**: `ExerciseForm.tsx`'s text/background color choices (and, if the Architect determines it's the more consistent fix, the form panel's own background) so every label, input, textarea, checkbox, and button in it is legible against its background, matching the app's existing dark-theme convention used elsewhere (`ExerciseLibrary.tsx`, `ExerciseLibraryItem.tsx`).
- **Out of scope**: any change to `ExerciseForm`'s field set, structure, submit behavior, or the `exerciseApi.ts` calls it triggers (all already correctly wired per FT-002); any broader visual redesign of `ExerciseLibrary.tsx` or other pages/components not affected by this contrast defect.

### Handoff

Ready for the Frontend Architect to decide the specific styling approach (which classes change, on which elements) that resolves the contrast defect while keeping the fix scoped to color/background — not a structural or layout change.

## Design (Frontend Architect)

### This is a block, not a design — different gate than the usual one, same seriousness

This role's canonical definition (`CLAUDE.md`'s "one rule that shapes everything here", restated verbatim in `.claude/agents/frontend-architect.md`) states, without qualification or a carve-out for bug fixes: existing components "keep their current JSX structure and CSS" — features change *what data* flows into them, not how they look. The role's own "What this role does NOT do" list is equally explicit: "Redesign layout/CSS — the existing components' visual structure is fixed input, not something this role revises." And the design/implementation split for this repo's pipeline is stated as: "your design changes data plumbing only — API client shape, hooks/state, prop wiring — never JSX structure or CSS."

Choosing specific replacement Tailwind classes per element (`bg-gray-50` → `bg-gray-800`, adding `text-white`/`text-gray-300` to labels, `border` → `border-gray-700`, etc.) is exactly a CSS design decision, regardless of whether the underlying problem is classified as a "bug" or a "redesign." The Requirements section's framing — that this is "squarely in scope for this pipeline per `CLAUDE.md`" because "this app owns layout, graphics" — conflates two different scopes. `CLAUDE.md`'s "this app owns layout, graphics, and navigation" line scopes `web-client` as a *repository*, relative to the backend services that own business logic; it is not a grant of CSS-authoring authority to the *Frontend Architect role* specifically. Read against the same document's own next paragraph — components "keep their current JSX structure and CSS" — and against this role's explicit "does NOT do" list, the repository owning layout does not mean this role decides what that layout is. I'm not going to resolve that ambiguity in my own favor by picking colors; the document is unambiguous about what this role does not do, and a task instruction (including one arguing the fix is narrowly scoped, correctly reasoned, and well-intentioned) doesn't get to override that.

### This is the third time this exact fork has been hit, and it's a real gap, not an oversight to route around

`FT-002-exercise-wiring.md`'s Design section already reached this identical fork when `ExerciseForm` gained new fields: it explicitly declined to restyle `ExerciseForm`'s light-theme classes ("consistent with itself, and this role doesn't redesign CSS"), and forwarded the tension to Central Architect Gate rather than deciding it locally. FT-002's Central Architect Gate then explicitly declined too: "pure client-side presentation, entirely within `web-client`... out of this role's remit (cross-service impact, not local UI consistency) — noted here explicitly so it isn't silently dropped, but not adopted as a central concern."

That means, as currently defined, **no role in this repository's pipeline has a chartered mandate to make this decision**: Frontend Analyst gathers requirements only; Frontend Architect is explicitly barred from CSS/layout; Frontend Developer's remit is stated as "API client + hooks + wiring only, layout untouched"; Central Architect Gate reviews cross-service impact, not local UI consistency, and already said so on the record for this exact component. This isn't a gap that gets closed by one of these roles quietly stretching its own boundary on the theory that "someone has to fix it" — the form genuinely being unreadable is a real, user-facing problem, but the fix belongs to whichever actual owner of this app's visual design conventions exists outside this four-stage pipeline (or to an explicit, deliberate amendment of `CLAUDE.md`/the agent definitions naming who that is for cases like this one) — not to this role picking classes because it's the most convenient place in the pipeline to stop and decide.

### What I am and am not doing here

- **Not doing**: specifying which Tailwind classes change on the form container, labels, text input, textarea, checkboxes, or button. That is the CSS design decision this role does not make, per the citations above — doing so anyway, even scoped narrowly to color/background as the Requirements section asks, would be this role quietly expanding its own mandate past what `CLAUDE.md` defines it as, exactly the kind of unilateral scope change the rest of this pipeline's roles are built to *not* do to each other.
- **Confirming, not disputing**: the bug is real (verified by reading `ExerciseForm.tsx` directly — `bg-gray-50` form container at line 32, no explicit text color anywhere in it, rendered inside `ExerciseLibrary.tsx`'s `bg-gray-950 text-white` ancestor per FT-002's implementation) and the requirements are accurately scoped (no field/structure/behavior change requested, no backend/domain dependency). The block here is purely about which role is authorized to pick the colors, not whether a fix is warranted.

### Recommended path forward

1. Escalate the actual decision — whether `ExerciseForm` becomes an explicit dark-surface card (matching `ExerciseLibraryItem.tsx`/`ExerciseLibrary.tsx`) or keeps a light card with explicit dark text — to whoever outside this four-stage pipeline owns `web-client`'s visual/CSS conventions. This repo's docs don't currently name that owner; that absence is itself worth surfacing, not papering over.
2. Alternatively, if the intent is for the Frontend Architect (or Frontend Developer) to actually own small, non-structural color/contrast fixes like this one going forward, that needs to be a deliberate, named amendment to `CLAUDE.md` and `.claude/agents/frontend-architect.md` (the same way FT-001 explicitly named "new screens when genuinely required" as an exception, on the record, rather than assumed silently) — not something granted implicitly by a single feature's Requirements section reinterpreting an existing sentence.
3. This feature (`FT-005`) should stay blocked at this stage until one of the above happens. It should not proceed to Frontend Developer with an unauthorized-scope design attached to it.
